# Spring Boot Kotlin Backend Skill Creation Plan

## Requirements Summary

Create a Codex skill that helps agents develop Spring Boot Kotlin backend projects according to the architecture and conventions in `moohee-lee/springboot-kotlin-skeleton`.

The skill must be version-independent. It may cite the skeleton's current stack as reference evidence, but it must not require Java 25, Kotlin 2.3.20, Spring Boot 4.0.5, Gradle 9.4.1, or jOOQ 3.19.31 unless a target project explicitly uses them. The README documents those as the reference repository's stack at lines 9-22.

The skill must cover:

- Hexagonal Architecture + DDD package boundaries: `adapter`, `application`, `domain`, `common`.
- New domain implementation flow: domain model, input port, output port, service, web adapter, persistence adapter, error code/message, schema/jOOQ regeneration.
- `common` package responsibilities: config, constants, enums, errors, exceptions, extensions, utils.
- i18n/message source structure, message resolver, and `MessageConverter` usage.
- Error handling with `ErrorCode`, domain exceptions, `DefaultException`, global error attributes, JSON error responses, and field-level validation errors.
- WebFlux + coroutine routing/handler conventions.
- Validation setup and request-binding helpers.
- WebClient configuration.
- jOOQ code generation, read/write datasource routing, transaction execution.
- Spring configuration processor and build configuration guidance.

Evidence from the reference repo:

- README states the project applies Hexagonal Architecture + DDD, WebFlux + Coroutine, error handling, i18n, and jOOQ at lines 3-5.
- README lists the full package tree at lines 30-143, layer roles at lines 148-191, and new-domain flow at lines 643-703.
- README summarizes `common` package contents at lines 707-717.
- `build.gradle.kts` declares WebFlux, WebClient, jOOQ, validation, configuration processor, and jOOQ generation at lines 43-60 and 72-154.
- `application.yml` configures message basenames, default locale, datasource split, and trace logging at lines 4-46.
- `WebFluxConfiguration.kt` configures message codes, validator, API versioning, and max in-memory size at lines 16-44.
- `GlobalErrorAttributes.kt` converts framework and custom exceptions into consistent error responses at lines 26-282.
- `DatabasePersistenceConfiguration.kt` configures read/write data sources, `LazyConnectionDataSourceProxy`, virtual-thread dispatcher, and read/write transaction templates at lines 19-69.
- `WebClientConfiguration.kt` configures timeout, codec size, and a shared `WebClient` bean at lines 10-29.
- `SampleHandler.kt` shows handler-side query binding, validation, enum path binding, required headers, body validation, and response creation at lines 16-85.

## Proposed Skill Shape

Skill name: `springboot-kotlin-backend`

Default creation path: `${CODEX_HOME:-$HOME/.codex}/skills/springboot-kotlin-backend`, unless the user wants it inside this workspace or another skills directory.

Resources:

- `SKILL.md`: concise workflow and reference navigation.
- `agents/openai.yaml`: generated by `init_skill.py` from deterministic display metadata.
- `references/architecture.md`: package structure, dependency direction, naming rules, and layer responsibilities.
- `references/domain-workflow.md`: step-by-step checklist for adding or changing a domain.
- `references/common-package.md`: `common` package inventory and what belongs or does not belong in each subpackage.
- `references/webflux-validation-error-i18n.md`: WebFlux router/handler pattern, validation, message resolver, i18n resources, error pipeline, response shape, trace ID handling.
- `references/persistence-jooq-transaction.md`: jOOQ, DDL generation, read/write datasource routing, transaction port/adapter, blocking JDBC with coroutine dispatcher.
- `references/build-configuration.md`: Gradle Kotlin DSL, version catalog/buildSrc guidance, configuration processor, WebClient dependencies, detekt/ktlint guidance.
- `scripts/check_structure.py` (optional but useful): lightweight project-structure checker that accepts a base package path and reports missing package families or common modules. Keep it advisory, not a generator.

Do not add a skill-level `README.md`, installation guide, changelog, or unrelated documentation. The skill-creator guidance explicitly keeps skills lean.

## Implementation Steps

1. Confirm skill name and destination.
   - Use `springboot-kotlin-backend` unless the user prefers a more architecture-specific name such as `springboot-kotlin-hexagonal`.
   - Ask before initializing if the skill should live outside `${CODEX_HOME:-$HOME/.codex}/skills`.

2. Initialize the skill with the system skill-creator script.
   - Command shape:
     ```bash
     /Users/moohee.lee/.codex/skills/.system/skill-creator/scripts/init_skill.py springboot-kotlin-backend --path "${CODEX_HOME:-$HOME/.codex}/skills" --resources references,scripts --interface display_name="Spring Boot Kotlin Backend" --interface short_description="Guide Spring Boot Kotlin backends with hexagonal architecture, WebFlux, error handling, i18n, jOOQ, and common package conventions." --interface default_prompt="Use $springboot-kotlin-backend when developing or reviewing a Spring Boot Kotlin backend that should follow the hexagonal architecture and common package conventions from the skeleton reference."
     ```
   - If skipping the checker script, initialize with `--resources references` only.

3. Write `SKILL.md`.
   - Frontmatter must include only `name` and `description`.
   - Description must include trigger contexts: Spring Boot Kotlin backend, hexagonal architecture, WebFlux/coroutine APIs, common package setup, error/i18n/validation/WebClient/jOOQ configuration, and new-domain implementation.
   - Body should stay under 500 lines and contain:
     - Quick workflow: inspect project, identify base package/build setup, choose domain boundary, add layer files, wire common infrastructure, verify.
     - Dependency direction rules: `adapter -> application -> domain`; `application` depends on ports/domain, not concrete adapters; `domain` stays framework-light or framework-free where feasible.
     - Reference navigation: tell agents which `references/*.md` file to open for each task.
     - Version policy: adapt APIs to the target project's Spring Boot/Kotlin versions; do not force skeleton versions.

4. Build `references/architecture.md`.
   - Include package tree and responsibility table derived from README lines 30-143 and 184-191.
   - Define allowed dependencies and anti-patterns:
     - Do not inject persistence adapters into handlers directly.
     - Do not put domain logic in routers/handlers.
     - Do not let `domain` depend on WebFlux, jOOQ, or Spring configuration.
   - Include base-package adaptation rules so `com.example.skeleton` becomes the target project's base package.

5. Build `references/domain-workflow.md`.
   - Convert README lines 643-703 into a domain-addition checklist.
   - Include concrete naming templates:
     - `domain/{domain}/model/{Entity}.kt`
     - `application/port/input/{domain}/{Domain}UseCase.kt`
     - `application/port/input/{domain}/model/*Command.kt`, `*Query.kt`
     - `application/port/output/{domain}/{Domain}Port.kt`
     - `application/service/{Domain}Service.kt`
     - `adapter/input/web/{domain}/{Domain}Router.kt`, `{Domain}Handler.kt`, `protocol/*Request.kt`, `*Response.kt`
     - `adapter/output/persistence/jooq/{domain}/{Domain}PersistenceAdapter.kt`
   - Add a required verification checklist for new domains: happy path, not found, validation failure, invalid enum/path/query, trace ID, read/write routing when persistence is touched.

6. Build `references/common-package.md`.
   - Document what belongs in:
     - `common.config`: WebFlux, error handler, ObjectMapper, trace logging, WebClient.
     - `common.constant`: shared constants like max buffer size, API version, default locale.
     - `common.enums`: `GenericEnum`, `DisplayEnum`, date pattern enum.
     - `common.errors`: `ErrorCode`, `CommonErrorCode`, domain error code enums, `ApiErrorResponse`, `ApiFieldError`, `ErrorSource`.
     - `common.exception`: `DefaultException`, validation exceptions, auth/permission exceptions, domain exceptions.
     - `common.extensions`: enum, `ServerRequest`, validator, Jackson, coroutine, collection, date/time helpers.
     - `common.utils`: message conversion.
   - Add guardrails against dumping domain-specific business logic into `common`.

7. Build `references/webflux-validation-error-i18n.md`.
   - Explain `coRouter` + `suspend` handler conventions using README lines 222-258 and `SampleHandler.kt` lines 16-85.
   - Document message code resolver and validator setup from `WebFluxConfiguration.kt` lines 16-44.
   - Document i18n basename and locale setup from `application.yml` lines 4-11 and README lines 443-475.
   - Document error response shape from README lines 298-396 and `ApiErrorResponse.kt` lines 3-19.
   - Document the global exception mapping from `GlobalErrorAttributes.kt` lines 45-282.
   - Include rules for field error source values: `BODY`, `QUERY`, `PATH`, `HEADER`.
   - Include trace ID behavior from README lines 526-542 and implementation references.

8. Build `references/persistence-jooq-transaction.md`.
   - Document jOOQ DDL-based generation from `build.gradle.kts` lines 72-109.
   - Document generated source registration and compile dependency from `build.gradle.kts` lines 111-154.
   - Document read/write datasource setup from `DatabasePersistenceConfiguration.kt` lines 22-68.
   - Document transaction port pattern and the reason blocking JDBC runs inside a dedicated coroutine dispatcher.
   - Include enum persistence guidance: store `GenericEnum.value`, convert DB values with `requireByValue`, treat invalid persisted enum values as server/data-integrity errors.

9. Build `references/build-configuration.md`.
   - Document required Gradle dependency categories, not exact versions:
     - Spring Boot WebFlux, Validation, jOOQ, WebClient/HTTP client, actuator if needed.
     - Kotlin reflection and coroutine/reactor bridge.
     - Jackson Kotlin module.
     - H2 or target DB runtime and jOOQ codegen dependencies.

10. Add optional `scripts/check_structure.py`.
    - Inputs: project root and base package path.
    - Checks: presence of core package families, `common` subpackages, resources folders, and build files.

## Acceptance Criteria

- `SKILL.md` has valid YAML frontmatter with only `name` and `description`.
- The skill triggers for Spring Boot Kotlin backend architecture work and does not trigger for unrelated Kotlin or Java tasks.
- `SKILL.md` stays concise and links directly to every reference file it expects agents to use.
- All detailed implementation guidance is in one-level `references/*.md` files, not duplicated heavily in `SKILL.md`.
- The skill explicitly states that skeleton versions are reference examples, not required target versions.
- The skill covers every requested topic: hexagonal package structure, `common`, i18n, message resolver, error code/exception/error response, WebFlux, validation, WebClient, jOOQ, Spring WebClient, and configuration processor.
- The domain workflow can guide a new domain from model through API, service, persistence, error message, schema, and verification.
- Error guidance produces the response shape `{status, code, message, path, traceId, errors?}` and field error source values.
- Persistence guidance separates ports from adapters and explains read/write transaction handling.
- `quick_validate.py` passes.
- If the optional checker script is included, it runs successfully on the reference skeleton and reports meaningful missing items on an incomplete fixture.

## Risks And Mitigations

- Risk: The skill overfits to the skeleton's exact versions.
  - Mitigation: Write all version-specific details as "reference repo example"; instruct agents to adapt API names to the target version.

- Risk: The skill copies too much source code and becomes bulky.
  - Mitigation: Use concise snippets and rules; link to reference filenames and patterns rather than pasting entire files.

- Risk: The skill encourages `common` package bloat.
  - Mitigation: Add explicit "does not belong in common" guidance and keep domain-specific logic inside domain/application packages.

- Risk: Spring Boot 4 API versioning does not exist in older projects.
  - Mitigation: Mark `ApiVersionConfigurer` as a version-dependent option and require fallback routing conventions for older Spring versions.

- Risk: jOOQ/JDBC is blocking while WebFlux is reactive.
  - Mitigation: Preserve the transaction port and dedicated dispatcher pattern; warn not to execute blocking JDBC on event loop threads.

- Risk: Error messages expose implementation details.
  - Mitigation: Keep client-facing messages from message resources; log internal details separately.
